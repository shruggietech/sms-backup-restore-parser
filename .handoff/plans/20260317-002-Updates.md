## Purpose and Context

<div style="text-align:justify">

This sprint cuts the first tagged release of sms-backup-restore-parser. The project is functionally complete for its initial scope: the streaming XML parser, CLI with parse/report/version subcommands, JSON Schema, test suite (47/47), documentation site, and PyInstaller build script are all in place. The sole remaining work is release-specific bookkeeping and the mechanical steps of building binaries, tagging the commit, and publishing the GitHub Release.

</div>

<div style="text-align:justify">

Binaries are built locally on each target platform (Windows and Ubuntu). There is no CI-based release pipeline for this release. The sprint plan provides explicit step-by-step guidance for each phase so that the process requires minimal improvisation.

</div>

---

## Implementation Ordering

<div style="text-align:justify">

Sections must be executed in strict sequential order. Section 1 (pre-flight verification) gates all subsequent work. Section 2 (changelog and metadata freeze) must complete before Section 3 (binary builds), because the version string baked into the executables must reflect the release state. Section 4 (git tag and GitHub Release) depends on the commit from Section 3. Section 5 (post-release housekeeping) runs last.

</div>

## 3. Build Standalone Binaries (Required)

### Description

<div style="text-align:justify">

Build the PyInstaller standalone executables for Windows and Ubuntu. Binaries are built locally on each target platform. PyInstaller does not support cross-compilation, so this section must be executed once on a Windows machine and once on an Ubuntu machine (or Ubuntu VM). The `dist/` directory is `.gitignored` and binaries are not committed to the repository; they are attached to the GitHub Release in Section 4.

</div>

### Implementation Instructions

A. On each target platform (Windows and Ubuntu), activate the virtual environment and install PyInstaller:

```bash
source .venv/bin/activate        # .venv\Scripts\activate on Windows
pip install pyinstaller
```

B. Run the build script:

```bash
python scripts/build.py
```

The script produces a single-file executable in `dist/`:

- Windows: `dist/sms-backup-parser.exe`
- Ubuntu: `dist/sms-backup-parser`

C. Verify the built binary works by running the version command:

```bash
# Windows
dist\sms-backup-parser.exe version

# Ubuntu
chmod +x dist/sms-backup-parser
./dist/sms-backup-parser version
```

Expected output: `sms-backup-parser 0.1.0`

D. Perform a basic smoke test by parsing a sample XML file (if available) with the built binary:

```bash
# Windows
dist\sms-backup-parser.exe parse sample-backup.xml -o ./smoke-test-output

# Ubuntu
./dist/sms-backup-parser parse sample-backup.xml -o ./smoke-test-output
```

If no sample XML file is available, the version check from step C is sufficient.

E. Collect the binaries from both platforms into a staging location for the GitHub Release. Suggested filenames for the release assets:

- `sms-backup-parser-v0.1.0-windows-x64.exe`
- `sms-backup-parser-v0.1.0-ubuntu-x64`

> **Note on platform coverage:** If only one platform is available at release time, ship what you have. A single-platform release with a note in the release body is acceptable for v0.1.0. The missing platform binary can be added to the release later.

### Affected Files

| File | Nature of Change |
|------|------------------|
| `dist/sms-backup-parser.exe` | Build artifact (Windows). Not committed. |
| `dist/sms-backup-parser` | Build artifact (Ubuntu). Not committed. |

### Acceptance Criteria

- At least one standalone binary is produced in `dist/`.
- The binary prints `sms-backup-parser 0.1.0` when run with the `version` subcommand.
- The binary is a standalone executable (no Python installation required on the target system).

## 4. Git Tag and GitHub Release (Required)

### Description

<div style="text-align:justify">

Tag the release commit, push the tag to the remote, and create a GitHub Release with the binaries attached. The release body contains a curated subset of the v0.1.0 changelog plus a link to the full changelog.

</div>

### Implementation Instructions

A. Ensure you are on `main` and the working tree is clean (the commit from Section 2 should be the HEAD):

```bash
git log --oneline -1
# Should show the "release: prepare v0.1.0 changelog and metadata freeze" commit
```

B. Create an annotated tag:

```bash
git tag -a v0.1.0 -m "v0.1.0 — Initial release

Streaming XML parser for SyncTech SMS Backup & Restore exports.
Converts SMS, MMS, and call log backups to schema-validated JSON
with built-in analytical reports."
```

C. Push the commit and tag to the remote:

```bash
git push origin main
git push origin v0.1.0
```

D. Create the GitHub Release. This can be done via the `gh` CLI or the GitHub web interface.

**Option 1: Using the `gh` CLI** (recommended if `gh` is installed):

First, create a release notes file. Save the following as `/tmp/release-notes.md` (or any temporary location):

```markdown
## Highlights

- **Streaming XML parser** that handles multi-gigabyte backup files with constant memory usage
- **Structured JSON output** with SMS, MMS (including parts and addresses), and call log records
- **Schema-validated** output conforming to a JSON Schema (draft-07) with full field documentation
- **Built-in reports** including summary statistics, per-contact breakdowns, and timeline views
- **Media stripping** to omit base64-encoded MMS media for compact analytical exports
- **Standalone executables** for Windows and Ubuntu (no Python required)

## CLI Usage

```
sms-backup-parser parse backup.xml -o ./output
sms-backup-parser report output/backup_sms.json -t summary
sms-backup-parser version
```

## Installation

Download the binary for your platform from the Assets section below, or install from source:

```
pip install -e .
```

See the [Installation Guide](https://shruggietech.github.io/sms-backup-restore-parser/installation/) for full details.

## Full Changelog

[CHANGELOG.md](https://github.com/shruggietech/sms-backup-restore-parser/blob/v0.1.0/CHANGELOG.md)
```

Then create the release:

```bash
gh release create v0.1.0 \
  --title "v0.1.0 — Initial Release" \
  --notes-file /tmp/release-notes.md \
  dist/sms-backup-parser-v0.1.0-windows-x64.exe \
  dist/sms-backup-parser-v0.1.0-ubuntu-x64
```

> Adjust the asset filenames to match whatever binaries were produced in Section 3. If only one platform binary is available, include only that one.

**Option 2: Using the GitHub web interface:**

1. Navigate to `https://github.com/shruggietech/sms-backup-restore-parser/releases/new`.
2. Select the `v0.1.0` tag.
3. Set the release title to `v0.1.0 — Initial Release`.
4. Paste the release notes content (from the template above) into the description field.
5. Drag the binary files from `dist/` into the asset upload area.
6. Leave "Set as the latest release" checked.
7. Click "Publish release".

None committed. The tag and release are remote-only artifacts.

- An annotated `v0.1.0` tag exists on the release commit.
- The tag is pushed to the `origin` remote.
